# Belief Building: B-1作成プロセス

## 概要

このルールは、ユーザーと**対話的に複数回やり取り**しながら、B-1（Belief-1）を作成するためのガイドです。

**重要**: B-1作成は一度で完結しません。ユーザーから段階的に情報を引き出し、丁寧に構築していきます。

## B-1作成の全体フロー

```
Step 0: プロジェクトを選択
    ↓
Step 1: 顧客の悩みを特定
    ↓
Step 2: 症状と影響を明確化
    ↓
Step 3: 仮想敵を設定
    ↓
Step 4: Beliefを定義
    ↓
Step 5: Claimを作成
    ↓
Step 6: Proofの方向性を決定
    ↓
Step 7: 2つのファイルを作成
    - 顧客の現状.md
    - Belief-1.md
```

## Step 0: プロジェクトを選択

### AIからユーザーへの質問

```
B-1（Belief-1）を作成します。

まず、どのプロジェクトで作業しますか？

プロジェクト名を教えてください。
（例：Contents Engine講座、○○講座、など）

または、プロジェクトのフォルダパスを指定してください。
（例：Outputs/Course/Contents Engine講座/Marketing/b1/）
```

### AIがすること

1. **ユーザーからプロジェクト名を受け取る**

2. **プロジェクトフォルダのパスを確認**
```
プロジェクト: {プロジェクト名}

出力先フォルダを確認しています...
Outputs/Course/{プロジェクト名}/Marketing/b1/

このパスで合っていますか？
```

3. **フォルダの存在確認**
   - フォルダが存在する場合 → そのまま使用
   - フォルダが存在しない場合 → 作成することを通知

```
✅ プロジェクトを確認しました

【プロジェクト情報】
- プロジェクト名: {プロジェクト名}
- 出力先フォルダ: Outputs/Course/{プロジェクト名}/Marketing/b1/

それでは、B-1の作成を開始します...
```

4. **プロジェクトルールファイルを読み込む**
```
プロジェクトの設計情報を確認しています...
.cursor/rules/{プロジェクト名}.mdc
```

## Step 1: 顧客の悩みを特定

### AIからユーザーへの質問

以前{プロジェクト名}/.cursor/rules/{プロジェクト名}.mdcにすでに、顧客のpainがあるので、それを抽出してユーザーに提示し、その上で悩みを詳細にする。

```
「B-1を作成します。まず、あなたの顧客（生徒さん）が抱えている悩みについて教えてください。

以下の質問に答えていただけますか？

1. 顧客はどんなことに困っていますか？
   （具体的に、複数あれば列挙してください）

2. その中で、最も深刻だと感じる悩みは何ですか？

3. その悩みを持つ顧客は、どんな言葉でそれを表現していますか？
   （もし顧客の声があれば、そのまま教えてください）
```

### ユーザーの回答を受けて

#### 1. まず回答を受け取る

「ありがとうございます。」

#### 2. 関連コンテキストを検索して提示（補完）

ユーザーの回答から**キーワード**を抽出し、以下を検索：

**A. Memoryから検索**
```
1. Persona Memory（persona-supermemory search）
   検索クエリ: {ユーザーの悩みのキーワード}

```

**B. Beliefs/フォルダから検索**
```
- grep を使用
- 検索対象: Beliefs/*.md
- 検索クエリ: {ユーザーの悩みのキーワード}
```

memoryやフォルダがヒットしなければ、ステップ4へ。


#### 3. 検索結果を提示（補完として）

もし関連する情報が見つかった場合：

```
実はこういうのもありましたよ：

📝 **あなたのメモリーから**:

【Memoryより】
- {結果の要約}

💡 **あなたの過去のBeliefから**:
- [[Belief - {タイトル}]]
- [[Belief - {タイトル}]]

これらも今回の悩みに関連していますか？
参考になりそうなものがあれば教えてください。
```

見つからない場合は、ステップ4へ。

#### 4. ユーザーに確認してブラッシュアップ

```
あなたが教えてくれた悩みと、見つかった情報を踏まえて、
今回B-1で扱う悩みを整理しましょう。

【現在の悩み】
{ユーザーが答えた悩みの要約}

【関連情報】（見つかった場合のみ）
- {MemoryやBeliefsから}

これで良いですか？
もし調整したい点や、追加したい視点があれば教えてください。
```

ユーザーの回答を受けて：
- **修正・追加の要望がある場合**: 内容をブラッシュアップして再提示
- **複数の悩みがある場合**: 「この中から、B-1で扱う**1つの悩み**に絞りましょう。どれが最も重要ですか？」
- **OKの場合**: 「ありがとうございます。それでは、次のステップに進みます。」

## Step 2: 症状と影響を明確化

### AIからユーザーへの質問

```
ありがとうございます。
次に、その悩みが引き起こす「症状」と「影響」について教えてください。

1. **症状（Symptom）**:
   その悩みを持つ人は、日常的にどんな症状を経験していますか？
   （例：時間がかかる、疲れる、ミスが増える、など）

2. **負の連鎖（Negative Ripple Effect）**:
   その悩みを放置すると、どんなまずいことが起きますか？
   （段階的に悪化していく様子を教えてください）

3. **最悪の結果（Worst Case）**:
   最悪の場合、どんな結果になってしまいますか？

4. **具体的なシーン（Movie Scene）**:
   その悩みを抱える顧客の「ある1日」を、映画のワンシーンのように
   描写できますか？
   
   例：
   - いつ？（時間帯、曜日）
   - どこで？（場所）
   - 何をしている？（具体的な行動）
   - どんな感情？（具体的に）
   - 何を考えている？（内面の声）
```

### ユーザーの回答を受けて

- 不足している要素があれば：「{要素}について、もう少し詳しく教えてください」
- 具体性に欠ける場合：「もっと具体的に、五感で感じられるような描写はできますか？」
- 十分な場合：「素晴らしいです。顧客の状況が鮮明に見えてきました。」
- 次のステップへ：「それでは、次のステップに進みます。」

## Step 3: 仮想敵を設定

### AIからユーザーへの質問

```
ここが重要です。

「その悩みは、誰のせいでもありません」
「顧客は悪くない」

ということを、メッセージとして伝えます。

以下の質問に答えてください：

1. **なぜ顧客は、その悩みを抱えることになったのですか？**
   （顧客の責任ではなく、外部要因は何ですか？）

   例：
   - 学校教育で教わらなかった
   - 社会では誰も教えてくれない
   - 業界の常識が間違っていた
   - 情報が溢れすぎている
   - 既存の方法論に限界がある

2. **何が「悪い」のですか？**
   （個人ではなく、社会や固定観念など）

```

### ユーザーの回答を受けて

- 個人攻撃になっている場合：「それは特定の人を傷つける可能性があります。もっと抽象的な『社会』や『固定観念』を仮想敵にしましょう」
- 適切な場合：「完璧です。『顧客は悪くない』が明確に伝わります。」
- 次のステップへ：「それでは、Beliefを定義しましょう。」

## Step 4: Beliefを定義

### AIがすること

これまでの情報をもとに、Beliefを提案：

```
これまでの情報から、以下のBeliefを提案します：

# Belief: {顧客が心から信じること}

例：
「守破離の守が大事なんだ」
「情報処理の問題が知的生産性を下げている」
「基礎を学ぶ機会がなかったんだ」

このBeliefで良いですか？
もし調整したい点があれば教えてください。
```

### ユーザーの確認を受けて

- 修正希望がある場合：修正して再提案
- OK の場合：「ありがとうございます。次にClaimを作成します。」

## Step 5: Claimを作成

### AIがすること

Beliefに基づき、Claimを提案：

```
Beliefに対する、あなた（販売者）の主張を作成します。

## Claim: {あなたの主張}

例：
「素直になれ」
「上流（信念）を整えるべき」
「基礎から学び直す必要がある」

**良いClaimの条件**:
- 1文で明確
- 断言形（「〜と思います」ではなく「〜です」「〜しろ」）
- 具体的な言葉

このClaimで良いですか？
もっと強い表現、または柔らかい表現に調整しますか？
```

### ユーザーの確認を受けて

- 修正希望がある場合：修正して再提案
- 複数のClaimが必要な場合：「Claim 2も作成しますか？」
- OK の場合：「ありがとうございます。最後にProofの方向性を決めましょう。」

## Step 6: Proofの方向性を決定

### AIからユーザーへの質問

```
Claimを裏付けるProof（証拠）を用意します。

3種類のProofがあります：

1. **Pathos（感情・ストーリー）**
   - あなた自身の失敗談や成功体験
   - 顧客のストーリー
   - 感情に訴える描写
   
   → 新規作成：「Story - タイトル」
   → 既存ファイル参照：接頭辞なしでそのまま

2. **Ethos（権威・データ）**
   - 専門家の意見
   - 研究データや統計
   - 論文や書籍の引用
   
   → 新規作成：「Ethos - タイトル」
   → 既存ファイル参照：接頭辞なしでそのまま

3. **Logos（論理・仕組み）**
   - なぜそうなのかの説明
   - ステップバイステップの解説
   - 仕組みの図解
   
   → 新規作成：「Logic - タイトル」
   → 既存ファイル参照：接頭辞なしでそのまま

質問：
それぞれのProofについて、以下の2つを教えてください：

1. **方針・説明**：どんな内容で証明したいですか？（普通の文章で）
   
2. **具体的なアウトプット**：
   - 新規作成するコンテンツ（Story、Ethos、Logicファイル）
   - または既存のファイルを参照

例：
【Pathos】
- 方針：上流設計がないと、魂が抜けた誰にも響かない文章が量産される
  - [[Story - ネタがないが故に、AIでコピペするしかない]]
  - [[10Xの企画で、悪徳商材屋ですか？と言われ、むかついた話]]

【Ethos】
- 方針：Chain of Beliefで5時間で1000名の募集枠が埋まった実績
  - [[Ethos - Chain of Beliefの成功事例]]

【Logos】
- 方針：上流を整えないと → メッセージがブレる → 響かない → 売上が立たない
  - [[Logic - なぜ上流設計が重要なのか]]
```

### ユーザーの回答を受けて

各Proofについて：
- **方針・説明**（文章のみ）
- **具体的なアウトプット**（Wikiリンク候補）

の両方をメモし、後でファイルに反映します。

## Step 7: 2つのファイルを作成

### AIがすること

これまでの情報をまとめて、**2つのファイル**を作成：

1. **顧客の現状.md** - 顧客の悩みと状況を記録
2. **Belief-1.md** - ClaimとProofを記録

---

### ファイル1: 顧客の現状.md

```markdown
---
mindmap-plugin: basic
---

# Belief: {定義したBelief}

## 顧客の悩み
{Step 1の内容を詳細に}


## 症状と影響
### 症状
{Step 2の症状を詳細に}

### 負の連鎖
{Step 2の負の連鎖}

### 最悪の結果
{Step 2の最悪の結果}

### 具体的なシーン
{Step 2のMovie Sceneを詳細に}

## 仮想敵
{Step 3の仮想敵}

### 顧客は悪くない理由
{Step 3の外部化理由}
```

---

### ファイル2: Belief-1.md

**重要**: Proofセクションは2層構造にすること
- 第1レベル：方針・説明文（Wikiリンクなし、普通の文章）
- 第2レベル：具体的なアウトプット（Wikiリンク付き）

```markdown
---
mindmap-plugin: basic
---

# Belief: {定義したBelief}

## Claim: {定義したClaim}

- **Proof (Pathos - ストーリー)**
    - {方針・説明文：どんな内容で証明するか}
        - [[Story - {タイトル}]]
        - [[{既存ファイル名}]]

- **Proof (Ethos - 権威・データ)**
    - {方針・説明文：どんなデータで証明するか}
        - [[Ethos - {タイトル}]]
        - [[{既存ファイル名}]]

- **Proof (Logos - 論理)**
    - {方針・説明文：どんな論理で証明するか}
        - [[Logic - {タイトル}]]
        - [[{既存ファイル名}]]

## Claim 2: {もしあれば}
...
```

**例**:
```markdown
## Claim 1: 一貫性のあるメッセージを上流で設計せよ

- **Proof (Pathos - ストーリー)**
    - 上流設計がないと、魂が抜けた誰にも響かない文章が量産される
        - [[Story - ネタがないが故に、AIでコピペするしかない]]
        - [[10Xの企画で、悪徳商材屋ですか？と言われ、むかついた話]]
     　     - [[Tweet - ネタがないが故に、AIでコピペするしかない]]

- **Proof (Ethos - 権威・データ)**
    - Chain of Beliefの成功事例：5時間で1000名の募集枠が埋まった実績
        - [[Ethos - Chain of Beliefの成功事例]]
        - [[Ethos - コンテキストエンジニアリングの重要性]]

- **Proof (Logos - 論理)**
    - 上流を整えないと → メッセージがブレる → 響かない → 開封率が落ちる → 売上が立たない
        - [[Logic - なぜ上流設計が重要なのか]]
        - [[Logic - 血の通った言葉の重要性]]
        - [[退職代行を使う人と、情報商材に騙される人は、同じ人である]]
```

## テンプレート: 質問の流れ

### 初回メッセージ

```
B-1（Belief-1）を作成しましょう！

B-1は、顧客の現状を理解し、深い共感によって信頼を構築する段階です。
ユーザーとの対話を通じて、以下を明確にしていきます：

0. プロジェクトの選択
1. 顧客の悩み
2. 症状と影響
3. 仮想敵（顧客は悪くない）
4. Belief（顧客が信じること）
5. Claim（あなたの主張）
6. Proof（証拠）の方向性

それでは始めましょう。

【Step 0: プロジェクトを選択】

まず、どのプロジェクトで作業しますか？

プロジェクト名を教えてください。
（例：Contents Engine講座、○○講座、など）
```

### 各ステップでの確認メッセージ

```
ありがとうございます。
{ユーザーの回答の要約}
ですね。

{必要に応じて追加質問}

確認できたら、次のステップに進みます。
```

### 最終確認メッセージ

```
素晴らしいです！B-1の構造が完成しました。
修正が必要な箇所があれば教えてください。

```
